The Challenge

Three pages. Seven tool files. Ten acceptance criteria.

Page 1
Home & Search
Search or select from ten addresses. Unknown address returns a helpful error.
✓ Already built
Page 2
Address Result
Address + map pin, then branch: active subscriber vs product choices.
→ Build it
Page 3
Details or Providers
Subscriber detail (active) or provider list for a chosen product.
→ Build it
  • 🚫
    De-identified sandbox. Never use real carrier names, logos, or process terms. Generic tech terms (FTTP, HFC, FTTN, FTTC, Fixed Wireless, speed tier) are fine.
  • 🚫
    No live integration. No login, accounts, payments, or internet calls. Mock data only.
  • 🔒
    Data is read-only. Never edit files in /data. Access everything through lib/data.js getters only.

Business Rules

The spec. Follow them exactly — these constraints are the point, not an obstacle.

  • 📍
    Status drives everything. active = subscriber. previous / never = non-subscriber. Never show network or service records for a non-active address.
  • 📡
    Technology drives products. Use Tool 2 technology to select the valid product set from Tool 3. Map pin uses Tool 1 coordinates.
  • ⬆️
    Eligibility drives the upgrade card. No upgrade card when upgrade_eligible is false. Provider count and names must match Tool 7 exactly.

The Seven Tool Files

Read-only mock data in /data. Always access through lib/data.js — never read or edit the JSON directly.

Tool 1
Locations
Address, coordinates, connection status for all ten locations
Tool 2
Technology
Access technology per location (FTTP, HFC, FTTN, FTTC, Fixed Wireless)
Tool 3
Products
Speed-tier catalogue by technology
Tool 4
Subscriber
Current product, connection date, upgrade eligibility flag — active only
Tool 5
Network
Health (RAG), sync speeds, last outage — active only
Tool 6
Service
Open tickets, appointments, health (RAG) — active only
Tool 7
Providers
Retail providers by technology
API contract: Worked endpoints: GET /api/locations, GET /api/locations/:id. All others return 501 with hints until you implement them: …/:id/technology, /api/technology/:tech/products, …/:id/subscriber, …/:id/network, …/:id/service, /api/technology/:tech/providers.

How the Sprint Runs

Six phases. Keep a working slice alive at every step — a small thing that runs beats a big thing that doesn't.

  • 1
    Read & Align
    Agree the stack, assign roles, pick demo addresses.
  • 2
    Plan
    Sketch pages, map rules, scaffold routes, draft tests.
  • 3
    Build Core
    Pages 1–2; connect Tools 1–6; status branch. Implement the six TODO endpoints in order: technology → products → subscriber (isActive guard!) → network → service → providers.
  • 4
    Build Detail
    Page 3 both modes (?mode=details and ?mode=providers). Connect Tool 7. Wire Back button to history.back().
  • 5
    Test
    Activate skipped tests in tests/api.test.js. Run path matrix in the browser. All 31 tests must pass before polish.
  • 6
    Polish & Rehearse
    Fix blocker defects only. Rehearse two journeys: Carlton (active) → Williamstown (never). Name your two 🔴 human moments.
GitHub arc: Issue → Branch → Explore → Plan → Build → Tests → PR → AI Review → 🔴 Human Merge. Before coding any issue: read it, confirm acceptance criteria, create a feature branch, then build.

Stuck? Use this.

An endpoint returns 501
It's still a TODO. Open server.js, find the matching stub, and implement it using the lib/data.js getter named in the hint comment.
Page 2 shows nothing
Open the browser console. If you see a network error on /api/locations/:id, the ?id= param is missing. Check result.js reads params.get('id').
Fixed Wireless shows 0 providers
You're using + encoding instead of %20. Change the fetch call to use encodeURIComponent(tech) — not a manual + replacement.
Subscriber data leaks for a non-active address
The isActive() guard is missing or in the wrong place. It must run before any call to getSubscriber / getNetwork / getService.
Upgrade card always shows (or never shows)
Check the condition: subscriber.upgrade_eligible === true. LOC-001 and LOC-004 are false; LOC-002 and LOC-003 are true.
Back button goes to Page 1
Replace location.href with history.back() in the back button's click handler.
No idea what to build next
Read the CLAUDE.md guardrails, then look at the six TODO stubs in server.js. They have hints. Pick the first 501 and implement it.
