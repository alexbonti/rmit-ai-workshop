Example output — Business Case
1. Problem Statement

nbn customers experiencing a service fault currently have no digital self-service option. The only path to lodge a fault report is to call the contact centre, where average hold times reach 23 minutes. Once a report is lodged, customers receive no proactive updates and must call back to check status — further driving inbound volume and customer effort.

Research across 1,840 residential customers (March 2026) surfaces three acute failure points: 84% of customers default to calling — not because they prefer the phone, but because no alternative exists. 7% give up entirely and wait for the fault to resolve itself. Customers without a fallback connection (22% of the cohort, predominantly rural and fixed-wireless) face the most friction.

2. Proposed Solution

Introduce a lightweight digital fault reporting tool, accessible via nbnco.com.au, that allows customers to log a fault, receive a reference number, and track status — without creating an account or calling the contact centre. This business case does not prescribe a technical architecture or vendor.

3. Key Benefits
BenefitDetail
Deflect inbound callsA 10% shift to digital self-service would reduce tens of thousands of inbound fault contacts per year.
Reduce repeat contactsProactive SMS/email updates and a reference number for self-serve tracking directly address the top frustration of having to call back.
Improve NPS for fault eventsCustomers want acknowledgement, a reference number, and a plain-English timeframe. A well-executed digital tool addresses all three.
4. Scope
✓ In scope✗ Out of scope
Address-based fault submission (no account required)
Fault type selection and optional description
Reference number generation and display
Status tracking via reference number
SMS and email status notifications
Mobile-first responsive design
WCAG 2.1 AA accessibility compliance		 Account creation or authenticated portal
RSP-facing tooling or back-office integration
Automated fault diagnosis or troubleshooting flows
MyNBN app changes (separate backlog)
Business customer fault reporting
5. Effort Estimate
WorkstreamSizeNotes
Discovery & UX designSAddress-lookup pattern already established. Design effort is incremental.
Front-end buildMMobile-first, WCAG 2.1 AA, no-account flow adds state management complexity.
API / back-end integrationMFault ingestion, reference number generation, status polling, notification dispatch.
Testing & accessibility auditSFunctional, regression, and WCAG 2.1 AA. Lightweight if built correctly from the outset.
OverallMEstimated 6–10 weeks end-to-end with a dedicated squad.
⚠ Primary risk — Connectivity during a fault event: 22% of customers have no mobile data fallback when their nbn service fails. A digital tool is unreachable by the customers who most need it. Mitigation: build for performance (sub-4-second load on 3G); the contact centre remains the safety net for this cohort.

UX Designer

You are a UX Designer working on the product described in the business case above. You have just reviewed the customer user research in this project files. Write 2 user stories for the primary end customer. Use the format: As a customer, I want to [action] so that [outcome].

Draw directly from the research — each story should trace back to a real frustration or need in the data. Include 2 acceptance criteria per story. Make them specific and testable — no vague words like 'quickly' or 'easily'.
Example output — User Stories
Story 1 — Reporting a fault without an account
Research source: 84% of customers call because no digital alternative exists. Customers ranked 'a way to report online without creating an account' as their top stated need.
"As a customer, I want to report an nbn fault using only my address and contact details so that I receive a reference number I can use to track progress without needing to log in."
  • 1A customer can submit a fault report — from landing on the page to receiving a reference number — in 5 or fewer form fields, with no account creation or login step at any point in the flow.
  • 2On successful submission, a reference number in the format NBN-2026-XXXXXX is displayed on screen and sent to the customer's nominated SMS or email address within 60 seconds.
Story 2 — Tracking status without calling back
Research source: Frustrations ranked #2, #3, and #4: no confirmation the fault was received; having to re-explain the problem if calling back; no way to check status without calling again.
"As a customer, I want to check the current status of my fault report using my reference number so that I don't have to call the contact centre and re-explain my problem to get an update."
  • 1A customer who enters a valid reference number on the status-check page is shown the current fault status, a plain-English description of what is happening, and an estimated resolution time — without being prompted to log in or provide any additional personal details.
  • 2When the fault status changes, the customer receives an SMS or email notification within 5 minutes of the status update occurring in the back-end system.

Review these user stories against the research data.

Example output — QA Note
✓ Story 2 — Clear winner

Story 2 directly targets the #1 ranked frustration: 23-minute hold times. If customers can self-serve a status check, the repeat call that drives hold time congestion is eliminated. Story 1 addresses the absence of a digital channel — a prerequisite — but the hold time pain is about what happens after a fault is lodged, not during initial report.

Action for BA: No change required. Recommend surfacing this rationale in sprint planning.

⚠ Both stories — Neither is resilient for the 22% cohort

Both stories require an active internet connection to function. 22% of customers — predominantly rural and fixed-wireless — have no mobile data fallback when their nbn service fails. Neither story includes any offline, SMS-in, or low-bandwidth fallback path.

Action for BA: Add a third user story covering the no-connectivity cohort, or explicitly document this gap as a known exclusion with a named mitigation.

✗ Gap — Frustration #5 is unaddressed

Frustration #5 — being told 'we'll call you back' with no timeframe — is not addressed by either story. This maps to a proactive outbound notification with a specific resolution timeframe, not just 'we'll contact you.'

Action for BA: Consider a third story, or revise Story 2 AC2 to specify that the notification must include an estimated resolution time.

✓ Criteria are clean — one flag to note

No vague language found. All four acceptance criteria include specific, measurable thresholds. One flag: Story 1 AC1 does not specify the network condition under which the end-to-end flow is measured. A pass on broadband may not reflect real-world behaviour for a customer on degraded mobile.

Action for BA: Add a test condition to Story 1 AC1: the flow must complete when tested on a throttled 3G connection at 1.5 Mbps.

Example output — Gap Analysis
GAP · 2 items unspecified

Notification content and mobile-first performance are promised but unspecified. The business case lists SMS and email notifications as in-scope and names mobile-first as a non-negotiable. Neither is adequately specified in the current stories. A notification that says only 'Your fault status has changed' technically passes Story 2 AC2 but fails the business case intent.

Action: Add a notification content specification and a mobile performance criterion specifying sub-4-second load on a throttled 1.5 Mbps 3G connection.

CLEAN · No out-of-scope items found

Both stories stay within the boundaries set by the business case. No authentication, no troubleshooting logic, no app-specific behaviour is implied by either story. The risk of scope creep is more likely to emerge during back-end integration design, not from the stories.

OPEN QUESTION · Blocks PRD finalisation

Does the fault management system expose a real-time API that this tool can consume — and who owns the integration? Every functional requirement in both stories depends on this. The reference number format, status update mechanism (poll vs. webhook), notification trigger, and delivery timeline cannot be specified until the API shape is confirmed.

Required action: Time-box a 3-day spike with Technology before PRD drafting begins. Output: written confirmation of API availability, update mechanism, reference number format, and integration team ownership.
